# Eventra Internal OS — Desktop Architecture (Tauri 2)

> Scope: this document covers **only** the desktop packaging of the private
> admin console (`apps/admin`). It does **not** touch Eventra Business
> (`apps/business`), Eventra Mobile/Consumer (`apps/consumer`), Shopify,
> Supabase, Railway, the design, navigation, metrics, calendar, backend or data
> model. The desktop app is a thin shell around the existing web frontend.

## 1. What was built

`Eventra Internal OS` is the existing `apps/admin` web SPA wrapped in a
[Tauri 2](https://v2.tauri.app) desktop runtime and packaged as a Windows
installer. The web app is unchanged: same React code, same routes, same
components, same services.

```
apps/admin/
├── src/                # UNCHANGED web frontend (Vite + React + React Router 7)
├── public/
├── index.html
├── vite.config.ts      # UNCHANGED
├── package.json        # + 3 scripts (desktop:dev/build/icon) + @tauri-apps/cli
├── dist/               # Vite production build (frontendDist for Tauri)
└── src-tauri/          # NEW — desktop runtime only
    ├── Cargo.toml
    ├── build.rs
    ├── tauri.conf.json
    ├── capabilities/default.json
    ├── icons/          # generated from the existing Eventra brand mark
    └── src/
        ├── main.rs     # thin launcher
        └── lib.rs      # window reveal, logging, plugins
```

## 2. Chosen architecture: **Option A — static frontend**

The audit (Bloque 0) confirmed that `apps/admin` is a **client-only SPA**:

| Property         | Value                                                        |
|------------------|-------------------------------------------------------------|
| Framework        | Vite 7 + React 18 + React Router 7 (`BrowserRouter`, SPA)   |
| Package manager  | npm workspaces (monorepo root `eventra-platform`)           |
| Dev command      | `vite` (Tauri runs it on a fixed port `1420`)               |
| Build command    | `vite build` → `dist/` (`index.html` + `assets/`)           |
| Output           | Static files, no server                                     |
| SSR / loaders / server actions / Prisma / Node routes | **None**               |
| Secrets client-side | **None** (`.env.example` documents `VITE_*` only)        |
| Backend          | External services (Supabase / Railway APIs), currently mock |

Because the admin has **no server-side runtime**, Tauri does **not** need to
spawn a local server (Option B was evaluated and rejected). Tauri simply serves
the static `dist/` bundle inside a native WebView2 window and the frontend talks
to the same external services it already uses on the web. This is the simplest,
most secure option: no local ports, no orphan processes, no health-check
sidecar, nothing new to keep in sync.

## 3. Runtime flow

```
Launch Eventra Internal OS
  → Tauri process starts, registers plugins (log, window-state, opener)
  → window-state restores previous size / position / maximized state
  → main window is created hidden (visible:false)
  → WebView2 loads the bundled dist/index.html (custom app protocol)
  → React (BrowserRouter) mounts and takes over routing client-side
  → on_page_load fires → window is shown + focused (no white flash)
  → app runs exactly like the web console; API calls go to Supabase/Railway
On close
  → window-state saves geometry
  → Tauri process exits cleanly (no child processes to reap)
```

A fallback timer reveals the window after 8 s if the page-load event is never
observed, so the app can never be stuck as an invisible process.

## 5. Identity & icons

- Product & window title: **Eventra Internal OS**
- Bundle identifier: **com.eventra.internal**
- Icons are generated by `tauri icon` from the existing Eventra brand mark
  (`apps/business/public/icons/icon-512.png`, the indigo calendar tile). No new
  logo was designed; the icon set in `src-tauri/icons/` can be regenerated at
  any time with `npm run desktop:icon -- <source.png>` from `apps/admin`.

## 6. What was deliberately NOT changed

- No redesign, refactor, new routes, new pages, new metrics.
- `apps/business` and `apps/consumer`: untouched.
- Supabase / Shopify / Railway config: untouched.
- Authentication: unchanged (see the security doc). The desktop build grants no
  automatic admin access.
- The only edits outside `src-tauri/` are additive npm scripts in
  `apps/admin/package.json` and the monorepo root `package.json`, plus the
  `@tauri-apps/cli` dev dependency.

## 7. Known limitations (pilot)

- Deep-link **hard refresh** (F5 on a nested route in the packaged app) depends
  on the app protocol serving `index.html` as a fallback; normal in-app sidebar
  navigation and relaunch (which start at `/`) are unaffected. See the build/report
  docs for the verified behavior.
- No auto-updater, no code-signing (`Unknown publisher` is expected and
  acceptable for private internal use). See `EVENTRA_ADMIN_DESKTOP_BUILD.md`.
